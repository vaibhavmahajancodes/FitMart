# Maintainer Scripts

This folder contains helper scripts designed to simplify and automate the pull request review and merge workflow for maintainers of the FitMart project.

These scripts are especially useful for maintainers who prefer reviewing pull requests locally and performing local merges instead of using GitHub's web-based merge options.

The workflow helps with:

* Faster PR review and merging
* Maintaining a clean Git history
* Creating proper local merge commits
* Preserving contributor attribution
* Generating reliable GitHub contribution activity for maintainers
* Avoiding repetitive Git commands

---

# Scripts Overview

## `review-pr.ps1`

This script automates the process of fetching and checking out a pull request locally.

### What it does

When a maintainer runs:

git reviewpr 

the script automatically:

1. Fetches the pull request branch from GitHub
2. Creates a temporary local branch using the PR number
3. Checks out that branch locally

This allows maintainers to:

* Review the code locally
* Run the project
* Test functionality
* Inspect changes properly before merging

### Example Workflow

Running:

git reviewpr 273

will:

* Fetch PR #273
* Create a local branch named `pr-273`
* Switch to that branch automatically

After this, the maintainer can test the pull request locally.

---

## `merge-pr.ps1`

This script automates the local merge workflow after a pull request has been reviewed and approved.

### What it does

When a maintainer runs:

git mergepr 

the script automatically:

1. Switches back to the `main` branch
2. Pulls the latest changes from the remote repository
3. Creates a local merge commit using the PR branch
4. Uses a GitHub-style merge commit message
5. Pushes the merge commit to the `main` branch
6. Deletes the temporary local PR branch

### Why this workflow is useful

Merging locally instead of using GitHub's web merge interface provides several advantages:

* Creates a real local merge commit
* Gives maintainers reliable GitHub contribution activity
* Preserves contributor authorship
* Maintains a clean and professional commit history
* Gives maintainers more control over the merge process

### Example Merge Commit Message

The merge commit generated by the script follows GitHub’s standard merge message format:

Merge pull request #266 from contributor-name/branch-name

This keeps the repository history clean and familiar.

# Recommended Setup (Preferred)

## Keeping Scripts Inside Repository

The recommended approach is to store the maintainer scripts directly inside the repository under:

```text
FitMart/
├── scripts/
│   ├── README.md
│   ├── merge-pr.ps1
│   ├── merge-pr.sh
│   ├── review-pr.ps1
│   └── review-pr.sh
```

These scripts are already included in the repository structure.

This keeps the tooling versioned alongside the project and makes it easier for maintainers to access, update, and share the scripts consistently across the team.

### Why this approach is preferred

Keeping the scripts inside the repository provides several advantages:

* Scripts stay version-controlled with the project
* Maintainers always use the latest script versions
* Setup becomes simpler for new maintainers
* Repository tooling remains centralized and easier to maintain
* Scripts can evolve together with the project workflow

As long as maintainers remain on branches that include the `scripts/` directory, the workflow operates smoothly.

# Setup Instructions

## Windows Setup

### Step 1 — Configure Git Aliases

Run the following commands from terminal or CMD:

```bash
git config --global alias.reviewpr "!powershell -ExecutionPolicy Bypass -File ./scripts/review-pr.ps1"
```

```bash
git config --global alias.mergepr "!powershell -ExecutionPolicy Bypass -File ./scripts/merge-pr.ps1"
```

---

### Step 2 — Use the Commands

Review a PR locally:

```bash
git reviewpr 273
```

Merge a PR locally:

```bash
git mergepr 273
```

---

## Linux / macOS Setup

### Step 1 — Make Scripts Executable

```bash
chmod +x scripts/review-pr.sh
chmod +x scripts/merge-pr.sh
```

---

### Step 2 — Configure Git Aliases

```bash
git config --global alias.reviewpr '!./scripts/review-pr.sh'
```

```bash
git config --global alias.mergepr '!./scripts/merge-pr.sh'
```

> Note: The merge script requires the GitHub CLI (`gh`) to be installed.

---

### Step 3 — Use the Commands

```bash
git reviewpr 273
```

```bash
git mergepr 273
```

---

# Important Note

Since the scripts are stored inside the repository itself, contributors' pull request branches may sometimes not contain the `scripts/` directory.

In such cases, the aliases may stop working after switching to the PR branch.

If this becomes an issue, maintainers can use the optional global scripts setup described below.

---

# Alternative Setup (Optional)

## Global Maintainer Scripts Folder

An alternative approach is storing the scripts outside the repository in a dedicated global folder.

Example:

### Windows

```text
D:\Projects\git-tools
```

### Linux / macOS

```text
~/git-tools
```

Copy the following files into that folder:

* `review-pr.ps1`
* `merge-pr.ps1`
* `review-pr.sh`
* `merge-pr.sh`

---

# Why some maintainers prefer this approach

When maintainers review a pull request locally, they switch to the contributor’s branch.

Some contributor branches may not contain the `scripts/` folder.

Keeping the scripts in a separate global folder ensures the commands remain available regardless of the currently checked-out branch.

This setup can be useful for maintainers managing multiple repositories or working across branches that may not always include the maintainer tooling.
